1. Azure Resource RBAC Policy: 관리그룹 및 구독에 계층형 구조에 따른 사용자권한 제어

  • RBAC관련 기본 내용 숙지
  • Builtin policy에 대해서 파악하고 대략적으로 어떤기능을 하는지 1문제
  • 권한체계 해석 하는것 1문제
  • 정리
    • Azure RBAC : 어떤 사용자가 어떤 리소스에 대해 어떤 작업을 수행할 수 있는지에 대한 권한을 정의하고 제어함.
    • Azure Policy : Azure 환경의 리소스가 조직의 표준이나 규정 준수 요구사항을 따르도록 강제하거나 감사함. (모든 가상머신에는 특정 태그가 무조건 포함되어야 한다거나, 특정지역에만 리소스를 배포하도록 강제하는 등)
    • Azure 리소스의 계층구조 : 테넌트 > 관리 그룹 > 구독 > 리소스 그룹 > 리소스
    • RBAC의 기본 개념:
      • 보안 주체(Security Principal): 액세스 권한을 부여받는 개체. Azure AD 사용자, 그룹, 서비스 주체(애플리케이션), 관리 ID 등이 될 수 있음.
      • 역할 정의(Role Definition): 수행할 수 있는 작업(Actions) 및 수행할 수 없는 작업(NotActions) 집합을 정의함. Azure는 소유자(Owner), 기여자(Contributor), 읽기 권한자(Reader), 사용자 액세스 관리자(User Access Administrator)와 같은 다양한 기본 제공 역할(Built-in roles)을 제공하며, 조직의 특정 요구사항에 맞춰 사용자 지정 역할(Custom roles)을 생성할 수도 있음.
      • 범위(Scope): 역할 할당이 적용되는 리소스의 계층적 수준. 관리 그룹(Management Group), 구독(Subscription), 리소스 그룹(Resource Group), 개별 리소스 등 4가지 주요 수준에서 범위를 지정할 수 있음. 상위 수준에 할당된 역할은 하위 수준의 모든 리소스에 상속됨.
  • 관련링크
    • Azure RBAC(Azure 역할 기반 액세스 제어)란?: https://learn.microsoft.com/ko-kr/azure/role-based-access-control/overview
    • Azure RBAC의 범위 이해: https://learn.microsoft.com/ko-kr/azure/role-based-access-control/scope-overview
    • Azure built-in role: https://learn.microsoft.com/ko-kr/azure/role-based-access-control/built-in-roles

  • 정리
    • 수명 주기 이벤트 발생 시점 및 데이터 삭제:
      • 수명 주기 관리 정책은 정의된 규칙(예: 마지막 수정/액세스 시간으로부터 며칠 경과)에 따라 데이터를 특정 계층으로 이동시키거나 삭제함.

    • 자동화 유지 관리 및 구성 방법:
      • 정책 규칙은 Blob 유형, 이름 접두사, Blob 인덱스 태그 등 필터를 사용하여 특정 Blob 집합에 적용가능.
      • 액세스 추적을 활성화해서 Blob의 마지막 액세스 시간을 기준으로 정책을 정의가능함.
    • Storage 계정 이벤트를 통해 자동화 하려면 Event Grid가 맞음.
      • Azure Event Grid : 이벤트 라우팅 서비스. 특정 이벤트를 구독하여 다르 서비스로 전달하는데 주로 사용됨.
      • Azure Event Hub : 데이터 스트리밍 서비스(kafka). 주로 빅데이터 수집 및 처리에 쓰임.

4. VNet 구성시 Inbound(LB), Outbound(NAT)구성방법과 기술구조/네트웍 흐름 이해

  • network flow에 대한 이해가 있어야함.
  • WEB/WAS에서 LB나 NAT 구성하는 방법 및 관련된 azure service를 알아야함.
  • NAT가 정확히 왜 필요한지 알아야 할것..
  • 정리
    • NAT Gateway: 서브넷과 연결되어 해당 서브넷의 모든 아웃바운드 트래픽에 대한 SNAT(Source Network Address Translation)를 제공. 고정된 공용 IP 주소를 통해 외부와 통신함.
      • 사용 목적 :
        • 보안: 내부 네트워크의 사설 IP 주소를 외부에 노출하지 않고, 하나의 공용 IP 주소로 통신하게 하여 내부 네트워크 비노출.
        • IP 주소 절약: 여러 개의 사설 IP 주소를 가진 내부 리소스들이 제한된 수의 공용 IP 주소를 공유.
        • 규정 준수/정책 적용: 특정 공용 IP를 통해서만 외부와 통신하게 하여 아웃바운드 트래픽에 대한 보안 정책을 중앙에서 관리가능.
    • Load Balancer: 웹/WAS 서버 앞에 위치하여 외부에서 들어오는 요청을 여러 웹/WAS 인스턴스로 분산
      • 사용 목적 :
        • 부하분산 및 고가용성(HA) 확보
      • 구성 요소 :
        • frontend IP : 외부에서 접속 가능한 IP
        • backend pool : 내부의 트래픽 분산 대상 서버(VM/VMSS) 그룹
        • ruleset : 특정 포트와 프로토콜로 들어오는 트래픽을 백엔드 풀의 서버로 어떻게 분산할지 정의
        • health probe :
    • 네트워크 흐름:
      • Outbound : 웹/WAS 서버(사설 IP) -> NAT Gateway/Load Balancer -> 공용 IP 주소 -> 외부 Target
      • Inbound : 외부 Source -> Load Balancer(ruleset filtering) -> 웹/WAS 서버(사설 IP)
    • WEB/WAS에서 LB나 NAT 구성 방법 및 관련된 Azure Service:
      • LB 관련 서비스:
        • Azure Load Balancer: TCP/UDP 트래픽을 분산하는 L4 로드 밸런서
        • Azure Application Gateway: HTTP/HTTPS 트래픽을 분산하는 L7 로드 밸런서, WAF기능 포함
        • Azure Front Door: 글로벌 트래픽 분산을 위한 CDN, WAF, SSL 오프로드 기능 등을 제공
      • NAT 관련 서비스:
        • Azure NAT Gateway: 서브넷 단위로 아웃바운드 SNAT 기능을 제공하는 서비스
        • Azure Load Balancer: 제한적으로 Outbound Connectivity를 지원하여 NAT기능을 지원(정적 port할당 및 NAT Gateway비해 우선권 낮음)
  • 관련 링크
    • Azure Virtual Network 설명서: https://learn.microsoft.com/ko-kr/azure/virtual-network/virtual-networks-overview
    • Azure Load Balancer 설명서: https://learn.microsoft.com/ko-kr/azure/load-balancer/load-balancer-overview
    • Azure NAT Gateway 설명서: https://learn.microsoft.com/ko-kr/azure/virtual-network/nat-gateway-resource

5. Saving Plan, Reserved Instance을 활용한 비용절감 방안 이해

  • Savings plan이랑 RI가 가능한 리소스와 불가능한 리소스에 대한 문제
  • 정리
    • Savings Plan (SP) : 1년 또는 3년 단위로 사용금액에 대해 계약하면 온디맨드 요금 대비 할인된 요금으로 제공.
      • 가능 리소스: Virtual Machines (VMs), App Service, Azure Functions, Azure Container Instances (ACI), Azure Dedicated Host 등
      • 불가능 리소스: 스토리지, 네트워킹, Cosmos DB 등 컴퓨팅 외 서비스
    • Reserved Instance (RI) : 1년 또는 3년 약정으로 특정 리소스에 대해 할인된 요금을 제공하는 약정 요금 모델.
      • 가능 리소스: Virtual Machines (VMs), Azure Cosmos DB, Azure SQL Database, Azure Synapse Analytics 등.
      • 불가능 리소스: App Service, Azure Functions, Azure Container Instances (ACI) 등.
    • 차이점:
      • Savings Plan: VM 유형이나 리전에 대해 유연함.
      • Reserved Instance: 특정 인스턴스 유형과 리전에 고정됨.
      • 사용할 리소스가 예측가능한 워크로드라면 RI가 가성비 좋음.
  • 관련 링크
    • Azure Savings Plans: https://azure.microsoft.com/ko-kr/pricing/offers/savings-plan-compute/
    • Azure Reservations (Reserved Instances): https://azure.microsoft.com/ko-kr/reservations/

6. Azure Storage Account 특징과 차이

  • 기술적인 level까지는 출제되지않음
  • crud 제약사항, 가용성에 대한부분. 예를들면 서비스 커버리지(region)에 따라 3가지로 나뉘는것들
  • 특징, 개념 등 어떤상품이 존재하고 어떤 상황에서 사용되는지에 대한 문제
  • 정리
    • Azure Storage 계정 유형:
      • 범용 v2 계정: 가장 많이 사용되는 유형으로, Blob, File, Queue, Table 등 모든 Azure Storage 서비스 지원.
      • 블록 Blob 계정: 블록 Blob에 특화되어 고성능 제공. Blob 개체 저장 시 사용.
      • 파일 스토리지 계정: Azure Files에 특화, 대규모 파일 공유 시 최적화된 성능 제공.
      • 프리미엄 페이지 Blob 계정 (Premium Page Blobs): Page Blob 전용, VM 디스크나 데이터베이스 트랜잭션 로그 등 고성능 I/O 용도에 적합한 SSD 기반 계정.
    • 복제 옵션:
      • LRS (로컬 중복 스토리지): 단일 데이터센터 내 3개의 데이터 복사본 유지. 가장 저렴하나, 데이터센터 장애 시 데이터 손실 위험.
      • ZRS (영역 중복 스토리지): 한 지역 내 3개의 가용성 영역에 데이터 복제. 데이터센터 장애로부터 보호.
      • GRS (지역 중복 스토리지): 주 지역과 수백 km 떨어진 보조 지역에 데이터 복제. 지역 전체 재해 대비.
      • GZRS (지역 영역 중복 스토리지): GRS와 ZRS를 결합, 주 지역 내 3개 가용성 영역에 복제 후 보조 지역에도 복제.
      • RA-GRS (Read-Access GRS): GRS에 보조 지역 데이터 읽기 권한을 추가한 옵션으로, 주 리전에 장애가 생겨도 보조 지역에서 읽기만 가능
      • RA-GZRS: GZRS에 읽기 접근을 추가한 옵션으로, 고가용성과 재해 대비 읽기 접근성을 모두 확보가능.
  • 관련링크
    • Azure Storage 계정 개요: Azure Storage 계정의 개념, 유형, 용도 등을 확인 https://learn.microsoft.com/ko-kr/azure/storage/common/storage-account-overview
    • Azure Storage 복제: 데이터 복제 옵션(LRS, ZRS, GRS, GZRS 등)에 대한 자세한 설명과 특징 https://learn.microsoft.com/ko-kr/azure/storage/common/storage-redundancy
    • Azure Storage 서비스 및 기능 요약: Azure Blob, Queue, Table, Files 등 각 스토리지 서비스에 대한 개요 https://learn.microsoft.com/ko-kr/azure/storage/common/storage-introduction

7. VMSS 특성에 대한 이해(aws autoscaling 이랑 같은 서비스)

  • VMSS사용에 대한 최근동향에 대해 숙지.

  • Autoscaling을 구현하기위해 VMSS를 사용하는데 이때 주의점 숙지

  • 너무 깊게까진 필요없지만 구성하는 순서라던지 LB로 어떻게 연결되는지 알아야함.

  • 예를들어 WebApp하나를 배포한다하면 어떤흐름으로 VMSS를 구성하는지 문제

  • 정리

      • VMSS사용에 대한 최근동향에 대해 숙지.

        • VMSS(가상 머신 스케일 세트)는 동일한 구성의 가상 머신(VM)을 그룹으로 관리하며, 규모에 따라 자동으로 VM 인스턴스 개수를 조정할 수 있습니다.
        • 기본 사용법:
          • Azure Portal, CLI, ARM 템플릿 또는 PowerShell을 이용해 VMSS를 생성할 수 있습니다.
          • 이미지, 인스턴스 크기, 디스크 옵션, 네트워크, 인증 정보 등을 설정합니다.
          • 운영 중에 인스턴스 수를 수동/자동으로 확장(Scale Out) 또는 축소(Scale In)할 수 있습니다.
          • VMSS 내 모든 인스턴스는 동일한 네트워크 및 부하 분산(LB) 구성에 연결할 수 있습니다.

      • Autoscaling을 구현하기위해 VMSS를 사용하는데 이때 주의점 숙지

        • Autoscaling(자동 확장) 시 주요 주의사항:
          • 상태 유지(Stateful) 서비스 지원 여부: VMSS는 기본적으로 Stateless(무상태) 웹/앱 워크로드에 적합합니다. 세션 유지나 데이터 로컬리티가 필요한 경우 별도 설계 필요.
          • 이미지 관리: VMSS 인스턴스는 동일한 이미지를 사용하므로, image를 변경(업데이트)할 때 전체 롤링 업데이트 방식 적용.
          • Auto-scaling Rule: CPU, 메모리, 네트워크 사용량 등 지표 기준으로 확장 규칙 설정. 과도한 Scale Out/In이 발생하지 않도록 쿨다운(Cooldown) 기간을 적절히 설정.
          • 기본 인스턴스 수 유지: 최소/최대/디폴트 인스턴스 개수를 정확히 설정하여 서비스 중단 방지.
          • 인스턴스 초기화 시간: VM 부팅 및 앱 배포에 소요되는 시간을 autoscale 정책에 반영해야 함.

      • 너무 깊게까진 필요없지만 구성하는 순서라던지 LB로 어떻게 연결되는지 알아야함.

        • VMSS 구성 순서

          1. 리소스 그룹 생성
          2. 가상 네트워크(VNet)와 서브넷 생성
          3. VMSS 생성
            • 이미지, VM 크기, 인증 방식 등 설정
            • 네트워크 인터페이스에 가상 네트워크 선택
          5. 확장 정책(Scaling policy) 설정
            • 자동/수동 인스턴스 조정 옵션 선택

          LB(Load Balancer) 연결 방법

          • VMSS 생성 중 '부하 분산 장치(LB)와 연결' 항목에서 기존 또는 신규 LB 선택
          • LB의 Backend Pool에 VMSS를 연결하여 트래픽이 자동 분산되도록 설정
          • 필요 시, LB에 Health Probe(상태 확인 프로브) 및 Load Balancing Rule(트래픽 라우팅 규칙)도 구성해야 함

      • 예를들어 WebApp하나를 배포한다하면 어떤흐름으로 VMSS를 구성하는지 문제

        • VMSS 인스턴스에 WebApp 런타임 환경 사전 설정(예: IIS, .NET, Node.js 등)
        • 배포할 WebApp(코드 및 라이브러리 등)을 VMSS 이미지에 포함하거나, CI/CD 파이프라인과 연계해 각 인스턴스에 배포 자동화 구성
        • LB(Layer 4/7)와 연결해 인바운드 웹 트래픽을 자동 분산 처리
        • 상태 무결성(Stateless) 구조 설계 권장(세션 유지 필요 시 Redis Cache 등 외부 저장소 활용)
        • 보안조치(SSL 인증서 설치/암호화, NSG 설정 등) 병행

8. DNS Resolver 구성법: AWS/Azure/Onpremise DNS동기화를 위한 구성과 필요한 Network 구성

  • 엔터프라이즈 환경에서는 internal로 보통 dns 구성을 하는데,, public으로 쓰려면 , 혹은 하이브리드로 할때 동기화 작업을 하게된다.

  • DNS관련 지식이 좀 있으면 좋지만 없어도 풀정도의 문제긴 함.

  • 문제의도는 DNS를 public - onprem간에 sync를 (어떤 방화벽이나 프로토콜 포트 등)어떤방식으로 해야되는지 나올것임.

  • 정리

    Public–On-prem DNS 동기화 방식(방화벽/프로토콜/포트 관점)

    1. 네트워크 연결 구성

    • VPN Gateway, 전용망(ExpressRoute/DirectConnect) 등을 이용해 Public 클라우드와 온프레미스 환경을 프라이빗으로 연결
    • 온프레미스 DNS와 퍼블릭 클라우드의 Private DNS 또는 DNS Resolver가 서로 통신할 수 있도록 사설 IP 대역을 지정

    2. 방화벽 및 네트워크 보안구성

    • 양방향 DNS 트래픽 허용 필요
      • 온프레미스 <-> Public DNS 리졸버(예: Azure DNS Private Resolver)
      • 반드시 신뢰된 DNS 서버 IP만 허용하도록 ACL 및 방화벽 정책 설정

    방화벽 규칙 (예시)

    | | | | | | -------------- | ------------------- | ------- | ---- | | 온프레미스 DNS | 클라우드 DNS 리졸버 | TCP/UDP | 53 | | 클라우드 DNS | 온프레미스 DNS | TCP/UDP | 53 |

    • DNS 동기화 및 쿼리, 포워딩을 위해 TCP와 UDP 모두 53번 포트를 방화벽에서 허용해야 함
      • UDP 53: 일반 쿼리 트래픽
      • TCP 53: Zone Transfer, 대용량 패킷 처리 등
    • 방화벽에서 DNS 서버의 IP 또는 VNet 서브넷 범위만 대상으로 정책 생성(불필요한 포트 오픈 방지)
    • 필요하면 SLAAC/IPv6가 활성화된 환경에서는 IPv6에 대한 포트/프로토콜도 반영 필요

    3. DNS 동기화 방법

    • 포워딩 또는 Conditional Forwarding 구성
      • 온프레미스 DNS에서 퍼블릭 DNS(클라우드 리졸버)가 관리하는 도메인을 포워딩
      • 클라우드 DNS 리졸버에서도 온프레미스 도메인에 대해 포워딩 룰 설정
    • 보안강화
      • DNS 트래픽 외 모든 인바운드/아웃바운드 트래픽 차단
      • DNS 서버 자체의 인증 및 트러스트 리스트 적용(Trusted IP 등)
      • 필요하면 TLS 기반의 DNS 통신(DNS over TLS/DoT 등) 고려

10. Azure 기반 Messaging 구성법 및 필요로 하는 Azure 서비스

  • 크게 4개(Event Grid, Event Hub, Service Bus, Relay) 가 있는데 대략적인 특징을 알고있으면 되는 개념적 문제

  • 정리

      • Azure Messaging은 시스템 간의 비동기 통신, 데이터 스트림 처리, 이벤트 송수신, 미들웨어 구축 등 다양한 목적으로 사용됩니다.
      • 기본 구성 절차:
        1. 요구 사항에 따라 적절한 메시징 서비스 선택(예: 대규모 이벤트, 트랜잭션 큐, 실시간 메시지 등)
        2. Azure Portal이나 CLI, ARM 템플릿 등을 활용해 서비스 인스턴스 생성
        3. 메시지 발신 및 수신 애플리케이션 개발(공식 SDK 및 API 지원)
        4. 보안(Shared Access Key, AAD 인증 등) 및 메시지 지속성 설정
        5. 필요 시, 모니터링/경고/재시도 등 운영 정책 적용

    | | | | --------------- | ------------------------------------------------------------ | | Event Grid | 대규모 이벤트 분배, 이벤트 기반 아키텍처 | | Event Hub | 실시간 이벤트/로그 스트림(빅데이터, IoT 등) 수집 | | Service Bus | 트랜잭션 메시지 큐, 복잡한 상호 응답 구조, FIFO, Dead-letter 지원 | | Relay | 온프레미스 앱을 클라우드에서 직접 호출(방화벽 친화적 통신) |

    Event Grid

    • 역할: 이벤트 기반 아키텍처를 위한 이벤트 라우팅 서비스

    • 특징

      • 이벤트 발행/구독 모델
      • Azure 리소스 상태 변화, 커스텀 이벤트, SaaS 이벤트 처리 가능
      • 수신자는 HTTP 웹후크 또는 Azure 함수, Logic App 등

      Event Hub

    • 역할: 대용량 실시간 데이터 및 이벤트 수집, 스트리밍 서비스

    • 특징

      • 초당 수백만 건 이상의 이벤트 처리 가능
      • 실시간 텔레메트리, 빅데이터 파이프라인, IoT 시나리오에 최적
      • 이벤트 분할(PARTITION) 및 소비자 그룹, 체크포인트 지원

      Service Bus

    • 역할: 신뢰성 높은 메시지 큐 서비스(엔터프라이즈 메시징)

    • 특징

      • FIFO(순서 보장), 세션, Dead-letter, 트랜잭션, 재시도 등 메시징 고급 기능 지원
      • Queue(점대점) 및 Topic/Subscription(퍼블리시-서브스크라이브) 모델 제공
      • 기업 내 애플리케이션 간 복잡한 업무 데이터 교환에 적합

      Relay

    • 역할: 네트워크를 통한 직접 연결을 지원, 온프레미스 호스트 노출 필요 없도록 중계

    • 특징

      • 하이브리드, 방화벽/프록시 친화적 통신(HTTP, TCP)
      • 클라우드에서 온프레미스 앱으로 직접 메시지 등 요청 전달 가능
      • 일반적으로 직접 연결이 불가능한 네트워크 간 안전한 통신 지원

    요약 표

    | | | | | ----------- | ------------------------ | ---------------------------------- | | Event Grid | 이벤트 배포, 라우팅 | Pub/Sub, HTTP endpoint, Azure 통합 | | Event Hub | 실시간 데이터 스트림 | 대용량 데이터, 파이프라인, IoT | | Service Bus | 트랜잭션 메시지 큐 | FIFO, Dead-letter, Topic 등 지원 | | Relay | 네트워크 직접 연결, 중계 | 온프레미스-클라우드 안전 연결 |

11. AKS Autoscale 이해: POD, Node(Node Group)

  • AKS 오토스케일링 관련 Container/Node 2가지로 나눠서 각각 어떤이벤트가 발생했을때 어떻게 처리하는지 알고있으면 되는 문제(기본적인 수준)

  • 정리

      • 1. AKS Autoscale로 Pod에 이벤트가 발생했을 때 처리 방법

        • Pod Autoscale(Horizontal Pod Autoscaler, HPA)란 워크로드의 메트릭(CPU, 메모리 등)에 따라 Pod의 복제 수(Replica)를 자동으로 조정하는 기능입니다.

        • 이벤트 발생 및 처리 흐름

          1. 메트릭 수집
            • Metrics Server, Azure Monitor 등에서 현재 Pod의 리소스 사용량(예: 평균 CPU 80% 이상) 모니터링
          2. 스케일 이벤트 탐지
            • 사전 정의된 임계치(Threshold) 초과 시 HPA가 스케일아웃(복제 증가) 또는 스케일인(감소) 이벤트를 감지
          3. 오토스케일러 동작
            • HPA Controller가 새로운 Pod 생성 또는 기존 Pod 제거 명령 실행
          4. 스케일 작업 반영
            • Kubernetes Scheduler가 적합한 노드에 신규 Pod 배치
            • Pod가 준비(Ready) 상태가 된 후 서비스에 트래픽 분배
          5. 지속적 모니터링 및 반복
            • 워크로드 변화에 따라 동적으로 Pod 수 조절 반복

        • 주요 고려사항

          :

          • HPA 정책의 타겟 메트릭과 쿨다운(Cooldown) 기간 적절히 설정
          • Pod이 필요 수만큼 빠르게 늘어나도록 이미지, 초기화 속도 최적화 필요

      • 2. AKS Autoscale로 Node에 이벤트가 발생했을 때 처리 방법

        • Node Autoscale(Cluster Autoscaler)란, 전체 클러스터 수준에서 스케줄 가능한 리소스(노드의 CPU/메모리)가 부족하거나 과잉 시 노드 풀 안의 VM(Node) 개수를 자동 조정하는 기능입니다.

        • 이벤트 발생 및 처리 흐름

          1. 스케줄 실패/리소스 부족 탐지
            • Pending 상태 Pod 또는 리소스 부족(노드 전체의 자원이 한계치 근접) 발생
          2. Cluster Autoscaler 동작
            • 신규 Pod을 수용할 여유가 없을 경우, Autoscaler가 VMSS(가상머신 스케일셋, Node Pool) 인스턴스 증가 요청
          3. 노드 증설 및 처리
            • 새로운 VM(Node)가 Azure에서 자동으로 프로비저닝
            • 새 노드가 클러스터 조인(Join), Scheduler가 Pending Pod을 이 노드로 배치
          4. 리소스 초과 시 노드 축소

          • 노드 생성에는 수 분이 걸릴 수 있으므로, 빠른 Auto Scaling을 위해 최소/최대 노드 수를 적절히 산정

20. Container 기본 개념 및 Dockerfile 이해

  • 예를들어 dockerfile 에서 아래 두개의 차이점은 ?
    • RUN apt update && apt install nginx
    • RUN apt update RUN apt install nginx
  • 두개로 레이어를 나누게되면 이미지 빌드 용량과 시간에 이득이 있을수는 있다
  • 하지만 docker cache때문에 apt update가 재수행되지 않는 치명적 단점이 있음

개념

  • Dockerfile에서 명령어를 한 줄(&&)로 묶느냐, 여러 줄(RUN 여러 개)로 나누느냐에 따라 이미지 레이어 생성 방식이 달라짐.
  • RUN apt update && apt install nginx
    • 하나의 레이어 생성 → apt update와 apt install이 한 번에 수행됨
    • 캐시 무효화 시 두 작업 모두 재수행됨
  • RUN apt update + RUN apt install nginx
    • 두 개의 레이어 생성 → Docker 캐시 때문에 apt update가 갱신되지 않을 수 있음(패키지 오래된 경우 문제 발생 가능)
    • 빌드 시간 단축 가능하지만 보안 패키지 업데이트에 취약
  • 실무에서는 apt update → apt install을 한 줄로 묶고 --no-cache 옵션 사용을 권장

9. DNS Resolver 방법 및 개념

  • Part1과 동일

  • 요약

    • Azure DNS Private Resolver는 Azure VNet 및 온프레미스 환경 간에 DNS 쿼리를 안전하고 효율적으로 처리할 수 있는 완전 관리형 서비스, 별도의 DNS VM 없이도 DNS 포워딩 설정이 가능.

    • Ruleset: VNet 내 리소스가 외부 DNS 또는 온프레미스 DNS로 특정 도메인을 포워딩할 때 사용
    • Ruleset을 설정해 특정 도메인을 특정 DNS로 라우팅 가능

10. Istio Service Mesh 개념과 각 자원/서비스 용어와 이해

  • gateway, virtual host 등 용어만 알면 풀수있는 문제.

    1. Istio Service Mesh 개념

    • Service Mesh란 마이크로서비스 환경에서 서비스 간 통신(네트워크 트래픽)을 관리·보호·관찰하는 인프라 계층입니다.
    • Istio는 대표적인 오픈소스 Service Mesh 솔루션으로, Kubernetes 등 분산 시스템 환경에서 서비스를 연결하고, 트래픽 제어, 보안, 모니터링을 일관되게 제공합니다.
    • Istio의 작동 방식:
      • 각 애플리케이션 Pod 옆에 "sidecar proxy"(Envoy Proxy)를 자동 배치
      • Istio Control Plane(istiod 등)이 전체 네트워크 트래픽 정책, 인증, 모니터링 등을 중앙에서 관리
      • 서비스 간 TLS, 라우팅, 감시(Observability) 기능을 sidecar proxy가 구현함

    2. Istio 주요 자원/서비스 용어 설명

    | | | | ---------------------- | ------------------------------------------------------------ | | Mesh | Istio가 관장하는 서비스 통신의 전체 네트워크(네트워크 구성 전체) | | Control Plane | Istio 전체 정책, 인증, 구성 등 식별/배포 역할(istiod) | | Data Plane | 실제 트래픽을 주고받는 sidecar proxy(Envoy)들이 배포되는 계층 | | Sidecar Proxy | 서비스(애플리케이션) Pod 옆에서 네트워크 트래픽을 대리(프록시) 처리(Envoy Proxy) | | Service Entry | 외부 서비스(클러스터 외부 포함)를 Mesh에 등록해 내부 라우팅 가능하게 하는 자원 | | DestinationRule | 서비스 목적지에 대한 트래픽 정책(예: TLS 필요 여부, LB 방식) | | VirtualService | 서비스 트래픽 라우팅 규정(경로, 헤더, 버전별 분배 등 상세 제어) | | Gateway | 메쉬 외부와 내부 간 트래픽 입출구 역할(HTTP, HTTPS, TLS 등), Load Balancer 역할도 포함 | | Envoy Filter | 추가 커스텀 트래픽 필터 및 동작 확장 | | PeerAuthentication | Service Mesh 내 "동료"(Pod 등) 간 인증 정책(OS, JWT, Mutual TLS) 관리 |

    3. Gateway, Virtual Host 등 용어 설명

    • Gateway
      • Istio Mesh에 외부(사용자, API, 클라우드 등) 트래픽을 받아들이는 진입점 역할
      • HTTP, HTTPS, TLS, TCP 등 프로토콜기반 트래픽을 제어·필터링
      • 예를 들면 웹 서비스에 퍼블릭 Load Balancer가 연결되는 것과 유사한 역할을 수행
      • Gateway 리소스와 VirtualService 리소스가 함께 사용되어 경로라우팅 정의
    • Virtual Host (Istio 용어로는 VirtualService의 host개념)
      • 하나의 Gateway가 여러 서비스(호스트, 경로)로 트래픽을 분기 가능
      • VirtualService는 host(도메인/서비스명)와 경로(match 조건)별로 트래픽 라우팅 규칙을 세분화할 수 있음
      • A/B테스트, Canary 배포, 라우팅 정책 변경 등 세밀한 트래픽 제어가 가능

    예시 구조 요약

    • Gateway: 외부 트래픽 진입점 → Istio Gateway 리소스 정의(포트, 프로토콜, 호스트 정보)
    • VirtualService: 들어온 트래픽을 host/경로/헤더 등 정책에 따라 Service나 버전별로 분기 처리

    요약 표

    | | | | -------------------- | ----------------------------------------- | | Service Mesh | 서비스 간 네트워크 트래픽 관리 구조 | | Control/Data Plane | 정책관리/트래픽 주고받기 | | Sidecar Proxy(Envoy) | 트래픽 프록시 기능, 서비스 옆에 자동 배포 | | Gateway | 외부 트래픽 진입점, L7 트래픽 제어 | | VirtualService/Host | 트래픽 상세 라우팅, 도메인·경로 기반 분기 | | DestinationRule | 트래픽 목적지 정책(LB, TLS 등) |

11. AKS 성능 개선에 대한 이해

  • container 성능개선을 위해 size, resource control, worker node 분배 등에 대한 문제

    1. 컨테이너 Size 조정

    • 적절한 이미지 경량화: 컨테이너 이미지를 최소한의 크기로(불필요한 라이브러리 제외) 빌드하면 배포/기동 속도가 빨라지고 OS/디스크 리소스 부담이 감소함

    • Pod/컨테이너 spec의 resource 설정

      • requests는 보장할 최소 리소스 (CPU, Memory)

      • limits는 사용할 수 있는 최대 리소스

      • 각 워크로드 특성에 맞게 신중하게 수치 선정(과소/과대 설정 방지)

      • 예시)

        yaml resources: requests: memory: "256Mi" cpu: "250m" limits: memory: "512Mi" cpu: "500m"

    • 다양한 크기의 컨테이너 분리 배포: 경량화된 마이크로서비스는 작은 리소스, 고성능 처리 서비스는 더 큰 리소스 요청으로 분리

    2. Resource Control 방법

    • 리소스 쿼터(Quota) 및 LimitRange 활용
      • 네임스페이스 단위로 리소스 한계값(LimitRange, ResourceQuota) 설정해 과다한 리소스 점유 앱 제어
    • HPA(VPA) Autoscaler 적용
      • Horizontal Pod Autoscaler(HPA): 동적 메트릭(CPU 등)에 따라 Pod Replica 수 자동 증가/감소
      • Vertical Pod Autoscaler(VPA): 애플리케이션의 실제 사용량 기반으로 Pod의 메모리/CPU 할당 동적 조정
    • QoS(Quality of Service) 클래스 고려
      • 컨테이너별 QoS(Resource QoS: Guaranteed, Burstable, BestEffort) 클래스 구분 및 중요 앱에 Guaranteed 적용

    3. Worker Node 분배/최적화

    • Node Pool 분리 & 노드 크기 선택

      • 부하 성격별로 별도의 Node Pool(예: 작은 VM Pool, 대형 VM Pool) 구성
      • 노드에 따라 VM 크기(메모리, CPU, 디스크/네트워크 사양) 다르게 구성
      • 예) 미션크리티컬 서비스: 고사양 노드, 일반 서비스: 경제적 노드

    • Affinity, Toleration, taint 활용

      • 특정 워크로드를 정해진 노드 Pool에 배치하도록 taint/toleration, affinity 규칙 사용

      • 예시)

        yaml nodeSelector: workload-type: high-memory

    • 분산 스케줄링 및 Drain/Upgrade 정책 관리

      • Pod이 특정 노드에 과다하게 몰리지 않게 스케쥴링 옵션 사용
      • 노드 장애·업그레이드 시, kubectl drain 등으로 Pod 재배치 전략 구성

    • 최소/최대 노드 Pool 사이즈 정책 설정

      • Auto Scaling 및 Scale-In 시, 노드 Pool 내 지나친 Scale Out/In 방지
      • 리소스 피크에 대비한 최소 Pool 크기 확보

    요약 테이블

    | | | | | ---------------- | ------------------------------------- | ------------------------------------- | | Size | 이미지 경량화, pod resource 튜닝 | 리소스 spec 설정, 서비스 분할 | | Resource Control | 리소스쿼터, HPA/VPA, QoS | 리소스 제한·분배 제어, Autoscale 적용 | | Worker 분배 | Node Pool 분리, Affinity/Taint/최적화 | 워크로드별 Pool 배치, 노드 크기 선택 |

12. Node Pool 개념 이해와 운영 방안 특징

  • 공통자원은 전용 노드 사용 해야한다. (필수는 아닌데 가이드라인)

  • system node pool, user node pool 두가지 있는데 특정 조치를 할때 어떤 노드풀에 넣을지 문제

    1. Node Pool 개념 이해

    • Node Pool이란 AKS 클러스터에서 VM(노드)들의 집합을 의미합니다.

    • 각 Pool은 서로 다른 VM 사양, OS 타입, 스케일(인스턴스 개수), 라벨 등을 가져서 다양한 워크로드 요구에 맞게 배포할 수 있음.

    • 하나의 AKS 클러스터에 여러 Node Pool이 존재할 수 있고, 각 Pool에 호환되는 Pod/워크로드만 배치 가능.

    • Node Pool의 주요 목적

      :

      • 서비스 특성에 따라 고사양/저사양 노드 분리
      • 운영 시스템과 사용자 앱(워크로드) 분리
      • OS/커널 버전 또는 보안 정책 별도 적용
      • 스케일링/유지보수/업그레이드 대상 분리 운용

    2. Node Pool 운영 방안 특징

    • 운영 용이성: 클러스터 유지, 노드 업그레이드, 장애 대응 시 각 Pool 별로 관리 가능
    • 유연한 배치: 워크로드별로 Pool을 라벨/taint/toleration/affinity 정책으로 구분하여, 적합한 리소스에 배치
    • 최적화된 비용 관리: 자원 요구에 따라 Node Pool VM 사양 조절(비싼 VM은 최소 Pool만 유지), 리소스 낭비 방지
    • 고가용성/보안: 미션크리티컬 서비스 Pool은 별도 관리(HA 구성), OS 보안패치 Pool별 적용 가능
    • 오토스케일링: 각 Node Pool 별로 별도 스케일 인/아웃 정책을 설정해, 트래픽/자원 상황에 따라 관리

    3. System Node Pool vs User Node Pool

    | | | | -------------------- | ------------------------------------------------------------ | | System Node Pool | AKS 클러스터의 핵심 시스템(필수 Infra Pod, kube-system, Metrics Server 등)이 반드시 배포되는 Node Pool - 최소 1개는 반드시 존재해야 하며, 주로 Cluster 관리 목적(작고 저렴한 VM 사용 권장) | | User Node Pool | 실제 비즈니스 워크로드(Pod), 사용자 앱, 마이크로서비스, 데이터처리 등 일반 서비스가 배포되는 Pool - VM 크기, OS, GPU, 스케일링 옵션 등을 자유롭게 지정 가능 |

    워크로드별 Node Pool 배치 실무 지침

    • System Node Pool에는 반드시 아래와 같은 AKS 관리/시스템 컴포넌트를 배포
      • kube-system 네임스페이스 Pod(코어 서비스)
      • 클러스터 모니터링, 네트워크, Ingress System 등 인프라 레벨 리소스
    • User Node Pool에는 다음을 배포
      • 실질적인 사용자 앱, 마이크로서비스, 백엔드 API, 데이터처리 등 업무 관련 Pod
      • 가중치 큰 리소스(예: GPU 활용 ML 워크로드), 특정 OS/커널 요구사항 등이 필요한 Pod

    특정 조치 예시

    • AKS 클러스터 업그레이드, 시스템 장애 복구 등 클러스터 관리 작업은 System Node Pool에 영향을 주지 않도록 사전에 백업/점검

    • 사용자 서비스에 오류, 리소스 소진 등 이슈 발생 시 User Node Pool만 증설/교체해 시스템 자원 분리 보호

13. CICD Pipeline 구성시 Credential관리 주의 사항에 대한 이해와 방법

  • 제목그대로 한문제

    1. 왜 Credential 관리가 중요한가?

    • Credential(인증서, 패스워드, API Key 등)가 노출되면 소스코드 유출, 서버 침해, 계정 탈취 등 심각한 보안 사고로 이어질 수 있음
    • 특히 CICD 환경에서는 빌드 서버, 테스트 환경, 배포 대상 등 여러 위치에 인증정보가 자동으로 전달/사용되어 노출 위험 증가
    • Credential이 잘못 관리되면 보안 취약점, 데이터 유출, 규정 위반(ISO, GDPR 등) 문제가 생길 수 있다

    2. Credential 안전 관리 방법

    (1) 소스코드에 Credential 절대 저장 금지

    • 패스워드, API Key, 인증서 등 환경 변수, 설정 파일 등으로 별도 분리
    • Git, SVN 등 형상관리 시스템에는 Credential이 들어간 파일 Push 금지

    (2) 비밀 관리 전용 솔루션 사용

    • 클라우드 Secret Manager와 Vault 사용
      • Azure Key Vault, AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault 등
      • CICD 파이프라인에서 API 또는 인증된 플러그인을 통해 비밀값을 호출
      • 비밀 추적 로그 및 접근 권한 정책 적용 가능

    (3) Credential 접근 권한 최소화

    • “필요한 서비스/권한만” 최소 범위로 적용(Principle of Least Privilege)
    • CICD 계정별 Role-based Access Control(RBAC) 강화

    (4) 암호화 및 전송 보안

    • Credential을 저장할 때는 강력 암호화 적용
    • 파이프라인 단계(빌드→배포)에서 Credential 전달 시 HTTPS/TLS 등 보안 채널 사용

    (5) 환경 변수로 전달

    • 빌드/배포 스크립트에서 환경 변수로만 Credential 전달
    • 환경 변수에 담긴 값은 실행 후 즉시 삭제, 로그에 노출 방지

    (6) 만기 및 교체 정책 운영

    • Credential 사용기간 만료(Expiration) 정책 수립
    • 정기적으로 키와 비밀번호를 변경하는 로테이션 작업 자동화

    3. 추가 보안 체크리스트

    • Credential Audit Log 및 접근 기록 반드시 관리
    • 접근 자동 탐지 및 경보(Alert) 설정(비인가 접근, 변경 감지 등)
    • 파이프라인 실패/오류 발생 시 Credential 자동 폐기(Leak 방지)

    정리표

    | | | | ----------------------- | ----------------------------------------- | | 소스코드에 저장 금지 | Config 분리, Git Ignore, Static Scan 강제 | | 비밀 관리자 사용 | Key Vault, Secrets Manager, Vault 등 이용 | | 접근 권한 최소화 | RBAC, 최소 권한 Role 설정, 권한 자동 만료 | | 암호화 및 채널 보안 | TLS, HTTPS, 강력 암호화 키 | | 환경 변수 전달 | 일회성 변수 적용, 로그 노출 방지 | | 주기적 키/패스워드 변경 | Rotation, 자동 교체 | | 감사로그 및 접근 기록 | Audit Log, Alert System 설정 |

14. Github Action의 기본구성 이해와 기본 Task명에 대한 이해

  • git action으로 실행할 수 있는 task 종류(상동)

    1. GitHub Actions 기본 구성 이해

    • GitHub Actions는 GitHub 저장소 내 이벤트(코드 Push, PR 등) 발생 시 자동으로 워크플로우(파이프라인)를 실행하는 CI/CD 서비스입니다.
    • 구성의 중심은 .github/workflows 폴더 속 YAML 파일입니다.

    기본 구성 객체

    | | | | ------------ | ----------------------------------------------------------- | | Workflow | 하나의 자동화 파일(예: build.yml) 전체/파이프라인 단위 | | Event | 트리거(trigger), ex) push, pull_request, schedule | | Job | 독립적 실행 단위(병렬/순차)로 여러 Step 묶음 | | Runner | 실제로 Job을 실행하는 컴퓨팅 리소스(공용, 자체 호스팅 가능) | | Step | Job 내에서 개별 명령/Task(스크립트, 액션, 셸 등) | | Action | Step에서 호출하는, 재사용 가능한 Task 함수(공식/커스텀) |

    Workflow 예시 구조

    yaml name: CI Pipeline on: [push, pull_request] jobs: build: runs-on: ubuntu-latest steps: - name: Checkout uses: actions/checkout@v4 - name: Set up Node.js uses: actions/setup-node@v4 - name: Install dependencies run: npm install - name: Run tests run: npm test

    2. 기본 Task명(주요 Action/Step)

    대표적으로 많이 사용되는 기본 Task(액션 혹은 Step 명칭):

    | | | | ----------------------------- | ---------------------------------------------------- | | checkout | 소스코드를 워크플로우 러너에 체크아웃함 | | setup-node | Node.js 개발 환경 준비 | | run | 지정한 스크립트(명령, Shell) 직접 실행 | | setup-python | Python 개발 환경 준비 | | actions/upload-artifact | 빌드 결과물(아티팩트) 업로드 | | actions/download-artifact | 이전 Job의 아티팩트 다운로드 | | setup-java | Java 환경 준비 | | actions/cache | 의존성/결과 etc 캐싱 | | deploy | 배포 명령(주로 커스텀 액션 혹은 배포 전용 스크립트) | | actions/checkout | 공식 소스코드 체크아웃 액션 | | actions/setup- | 특정 언어/환경 준비(예: setup-node, setup-python 등) |

15. Storage의 암호화 방식에 대한 이해

  • 검색잘하면 맞출만한 문제, 함정 느낌 살짝?

  • DB컬럼 암호화 같은게 섞여있는데 그게 .. 틀린것

    1. Azure Storage 주요 암호화 방식

    (1) 서버측 암호화(Server-Side Encryption, SSE)

    • 데이터가 Blob, File, Queue, Table 등 저장소에 저장될 때 자동으로 암호화됩니다.
    • Microsoft에서 기본적으로 AES-256 대칭키로 각 데이터를 암호화
    • 애플리케이션 또는 사용자가 별도 암호화 코드를 작성하지 않아도 됨
    • 복호화는 Azure Storage가 자동 처리해 투명하게 제공

    암호화 키 관리

    • 기본 관리형 키(Microsoft Managed Keys)
      • Azure가 직접 키를 생성·관리(자동 회전 및 보안 관리)
    • 고객 관리 키(Customer Managed Keys, CMK)
      • 사용자가 Azure Key Vault 등에서 직접 키를 생성하여 Storage Account에 지정
      • 키의 주기적 교체, 삭제, 권한 제어 등 추가 보안 정책 적용이 가능

    (2) 전송 데이터 암호화(In-Transit Encryption)

    • 클라이언트와 Azure Storage 간의 데이터 송수신은 TLS/SSL(HTTPS)로 암호화되어 전송됨
    • 인증서, HTTPS 필수 설정으로 패킷 가로채기 방지

    (3) 클라이언트측 암호화(Client-Side Encryption) (선택적 적용)

    • 사용자가 파일, 오브젝트를 저장하기 전에 직접 암호화하여 업로드하는 방식
    • 복호화 키는 Azure가 보관하지 않고, 고객이 자체적으로 관리
    • 강력한 보안이 필요할 때 별도로 활용(적용·관리는 고객 책임)

    2. 서비스별 암호화 적용지원

    | | | | | | ------------ | ---- | ---- | ---- | | Blob Storage | O | O | O | | File Storage | O | O | O | | Table/Queue | O | O | O |

    3. 암호화 설정 예시 및 운영 포인트

    • 기본적으로 Azure Storage Account를 생성하면 SSE가 자동 적용되어 별도 설정 불필요
    • CMK를 사용할 때는 Azure Key Vault와 연동(역할 권한, 키 주기 관리 등) 필요
    • Storage Account 정책에서 HTTPS만 허용하도록 "Require secure transfer" 옵션 설정
    • 규정 준수 강화를 위해 암호화 감사 로그, 키 접근 권한 정책을 병행하여 운영

    요약

    | | | | ------------ | ----------------------------------------------- | | Server Side | 데이터 저장 시 자동 AES-256 암호화, 복호화 투명 | | Customer Key | Key Vault 등 고객이 직접 관리(교체/권한/폐기) | | In-transit | TLS/SSL(HTTPS)로 전송 데이터 암호화 | | Client Side | 업로드 전에 직접 암호화, 키 자체 관리 |

    • CMK 설정 가이드 https://learn.microsoft.com/ko-kr/azure/storage/common/customer-managed-keys-overview

16. Azure Native 보안 서비스 이해

  • 어떤 서비스가 있는지 알면 되는 문제

    Azure Native 보안 서비스 주요 종류 및 역할

    1. Azure Active Directory (AAD)

    • Azure 기반의 ID 및 액세스 관리(SSO, MFA, RBAC, Conditional Access 등)를 담당
    • 클라우드 앱, 온프레미스 리소스 통합 인증/권한 통제 핵심 서비스

    2. Microsoft Defender for Cloud

    • 클라우드 환경의 통합 보안 posture 관리, 위협 감지, 자동 정책권고 기능 제공
    • 서버, VM, Storage, DB, 컨테이너 등 다양한 서비스 대상으로 취약점·위협 탐지 및 대응

    3. Azure Key Vault

    • 비밀값(Secret), 인증서, 암호화 키 전용 저장소 및 관리 서비스
    • 보안 권한, 키 라이프사이클(회전·폐기), 감사로그 등 완전관리형 지원

    4. Azure Firewall

    • Azure Virtual Network 내에서 동작하는 완전 관리형 네트워크 방화벽
    • 트래픽 필터링, 앱/네트워크 룰 제어, 위협 인텔리전스 기반 차단 지원

    5. Azure DDoS Protection

    • 인터넷으로 노출된 Azure 리소스에 대한 DDoS(분산 서비스 거부 공격) 방어 자동화
    • 실시간 트래픽 모니터링 및 자동 완화

    6. Azure Security Center (Defender for Cloud에 통합)

    • 보안 정책 설정, 컴플라이언스, 취약점 평가, 자동 권고 기능

    7. Azure Policy

    • 구성 정책/규정 준수 자동 적용 및 평가 도구
    • 리소스 배포/관리 시 표준 규칙 적용(예: Storage 암호화, 네트워크 제한 등)

    8. Azure Sentinel

    • 클라우드 기반 SIEM(이벤트/로그 통합 분석) 및 SOAR(자동 사고 대응) 서비스

    요약표

    | | | | ------------------ | ----------------------------------------- | | Azure AD | 인증/권한/SSO/MFA 관리 | | Defender for Cloud | 통합 보안 posture, 위협 탐지, 권고 자동화 | | Key Vault | 암호화 키, Secret, 인증서 완전관리형 저장 | | Azure Firewall | 네트워크 트래픽 필터링, 룰 제어 | | DDoS Protection | DDoS 공격 방어 자동화 | | Policy | 표준 정책 적용, 규정 준수 자동 점검 | | Sentinel | SIEM/SOAR 통합 관리 |

    특징

    • 완전관리형/자동화: 별도 설치 없이 Azure 포털/콘솔에서 바로 설정·운영
    • 클라우드/온프레미스/하이브리드 지원: Azure 자원뿐 아니라 외부 리소스까지 통합 보안 관리
    • 보안 규정준수, 레포팅, 인텔리전스 내장

18. Migration From VMWare VM: 제약조건, 구성법 등

  • 기술적 요구사항 중 데이터플랫폼의 특징만 알면 된다고 하심.

  • 함정 문제 느낌

    VMware 기반의 온프레미스 인프라를 Azure 환경으로 이전(Migrate) 하는 방식. 주로 Azure Migrate를 통해 전체 워크로드(Compute, Network, Storage, App 등)를 이전.

    🔧 구성 방식

    1. Azure Migrate 프로젝트 생성
      • 리전 선택, Assessment & Migration 툴 연계 가능
    2. Appliance 설치 (VMware 환경 내)
      • OVA 파일 다운로드 후 vCenter에 배포
      • Inventory 수집, 디스크/네트워크 정보 확보
    3. Assessment 수행
      • 성능 기반 크기 조정, 비용 추정, 호환성 확인
    4. Migration 수행
      • agentless vs agent 기반 마이그레이션 선택
      • 복제 설정 → 복제 완료 → 마이그레이션 트리거

    📌 기술적 요구사항

    | | | | -------------- | ------------------------------------------------- | | vCenter Server | 6.7 이상 권장, 연결 필수 | | VMware Tools | 최신 버전 설치 권장 | | 포트 | TCP 443, 9443, 44368 (Agentless), 22/3389 (Agent) | | 계정 | vCenter 연결을 위한 관리자 권한 계정 | | OS 지원 | Windows Server 2008 이상, Linux Kernel ≥ 2.6 | | 리소스 | Azure Network/Subnet/VNet 사전 구성 필수 |

    ⚠️ 예상되는 함정/주의 포인트

    • Agentless 방식: OS 지원에 제한 있음 (Linux 일부 미지원)
    • Disk Mapping 문제: LVM, SW RAID가 있으면 평가만 되고 마이그레이션 불가
    • Shutdown 없이 복제 가능하지만, Cutover 시 다운타임 있음
    • NSG, UDR 구성 사전 고려 필요 (DNS, 보안 등)

    ✅ 장단점

    | | | | ---------------------------------- | ---------------------------------- | | 무중단 복제 (복제 후 전환) | Cutover 시 다운타임 발생 | | 성능 기반 사이징 및 비용 산정 제공 | 복잡한 네트워크는 수동 재설정 필요 | | 대규모 워크로드 처리 지원 | OS/디스크 구조 제약 존재 |

    🧠 자주 나오는 개념 & 포인트

    • Replication Appliance는 VMware에 설치됨
    • Azure Migrate는 복수의 도구 통합 플랫폼 (Discovery, Assessment, Migration)
    • 평가 기반 권장 VM 사이즈 제공 → 비용 절감 전략 포함
    • 앱 단위 전환은 App Containerization, refactor 필요

    📄 공식 문서 링크

    • Azure Migrate Overview
    • VMware to Azure Migration
    • 지원 OS 목록

    ⏱️ 초압축 요약

    • Azure Migrate → vCenter 연동 + Assessment + Replication + Cutover
    • Agentless 복제는 제한적 → OS, 디스크 구조 확인 필요
    • 사전 준비 항목: vCenter, 포트, 계정, Azure Network
    • 복제는 무중단, Cutover는 다운타임 있음

19. PV/PVC with Azure File 구성시 필요한 Network 설정

  • private endpoint 알면 풀정도 문제

    📘 개념 정의

    • Azure Files: Azure에서 제공하는 SMB 기반의 파일 공유 서비스
    • PVC (PersistentVolumeClaim): 쿠버네티스에서 상태 저장 볼륨을 요청하기 위한 객체
    • CSI Driver: Azure Files를 쿠버네티스에서 연동하기 위한 드라이버 (azurefile-csi-driver)
    • Private Endpoint: Azure Storage에 대한 전용 네트워크 접속(Private IP 기반)

    🔧 구성 방식

    1. Storage Account 생성

      • Premium/Standard 선택
      • SMB 3.0을 지원하도록 설정

    2. Private Endpoint 연결

      • Storage 계정에 Private Endpoint 추가 (파일 서비스용)
      • 연결된 VNet의 Subnet에 IP 할당됨
      • privatelink.file.core.windows.net DNS 사용

    3. Private DNS Zone 구성

      • privatelink.file.core.windows.net Private DNS Zone 생성
      • Virtual Network에 링크 필요
      • 자동 등록된 A 레코드 존재 여부 확인

    4. PVC 생성 방식

      • Storage Class에서 csi.azure.com을 정의

      • Storage Account key 또는 managed identity 기반 인증

      • 예시:

        kind: StorageClassparameters:  skuName: Standard_LRS  secretName: azure-secret

    🚧 제약조건 및 확인사항

    | | | | ------------- | --------------------------------------------------------- | | DNS 설정 필수 | Private DNS Zone 연결 누락 시 내부 연결 실패 | | 인증 이슈 | Identity 사용 시 CSI 버전 요구, Key 방식은 비밀 관리 필요 | | Subnet NSG | NSG에 Storage 접근 포트 허용 필요 (TCP 445 등) | | Zone 제한 | 일부 구역에서 Azure Files Private Endpoint 미지원 |

    ✅ 장단점

    | | | | ----------------------- | ----------------------------------------------- | | 공용 인터넷 완전 차단 | 설정 복잡, DNS 연동 필수 | | 보안 강화 (내부망 전용) | CSI 버전 의존, 마운트 실패 시 트러블슈팅 어려움 |

    🔐 보안 고려 사항

    • Azure Policy로 외부접속 허용 금지 정책 설정

21. 구독 할당 및 관리에 대한 이해

  • Account policy 에 따라서 A라는 사용자가 특정 action을 할수있냐없냐

  • 특정 action을 수행하려면 어떤 policy를 부여해야하는지 그런 문제

    1. 개념 정의 / 설명

    • 구독(Subscription): Azure 리소스 사용의 청구 및 정책 단위
    • 관리 그룹(Management Group): 구독을 그룹화하여 정책 적용 단위 제공
    • RBAC(Role-Based Access Control): 역할 기반 권한 관리 체계

    2. 개념 요약

    • 계층: Management Group > Subscription > Resource Group > Resource
    • 역할: Owner, Contributor, Reader, Custom Role
    • 정책: Azure Policy를 통해 거버넌스, 규정 준수

    3. 구성 방식

    1. 관리 그룹 구조 설계: 조직 단위에 맞춘 계층 구성
    2. 권한 할당: 구독/리소스 그룹 수준에서 역할 부여
    3. Custom Role: JSON 정의로 세부 권한 제어
    4. Azure Policy: 정책 할당으로 규정 준수 강제

    4. 제약 조건 / 확인 사항

    • 역할 상속(Inheritance)과 우선순위 이해
    • 구독 수당 Azure Policy 할당 제한(기본 1000)
    • 관리 그룹 수준 Custom Role 지원 여부
    • MFA, 조건부 액세스 정책 확인

    5. 장단점

    | | | | ----------------------- | ----------------------------- | | 세분화된 권한 분리 | 스코프 복잡 시 관리 어려움 | | 거버넌스 정책 일관 적용 | Custom Role 관리 오버헤드 | | 중앙 집중식 정책 제어 | 잘못된 정책 설정 시 전체 영향 |

    6. 보안 고려사항

    • 최소 권한 원칙 적용
    • 조건부 액세스: MFA, 지리적 접근 제한
    • 모니터링: Azure Activity Log, Sentinel 연동
    • 비용 관리: Budgets, Cost Management 설정

    7. 공식 문서

    • Azure RBAC 개요
    • Azure Policy 개요
    • Management Group 소개

    8. 시험 대비 핵심 포인트

    • 역할 스코프 유효 범위
    • Policy vs RBAC 차이
    • Management Group 최대 계층 깊이
    • Custom Role JSON 구조
    • 특정 사용자가 특정 액션 가능 여부 → 역할 + 스코프 조합으로 판단

22. Vnet Flow log 서비스 종류와 특징 이행

  • 최근 새로운 서비스인 Vnet flow log 가 출시되어서 기존있던거랑 신제품의 차이 알면되는 문제

    VNet Flow Logs는 Azure Virtual Network(VNet) 수준에서 발생하는 트래픽 흐름을 기록하고 분석하기 위한 기능입니다. 기존 NSG(Network Security Group) Flow Logs의 한계를 보완하며, 네트워크 가시성 확보 및 보안 분석에 활용됩니다.

    🆕 기존 NSG Flow Logs vs. VNet Flow Logs (2023~2025 주요 변화)

    | | | | | ----------------- | ----------------------------------- | ------------------------------------------- | | 수집 단위 | NSG (Network Security Group) 단위 | Virtual Network 전체 | | 데이터 가시성 | 제한적 (NSG 기준 Inbound/Outbound) | 전체 흐름 기반 (암호화 여부 포함) | | 구성 | NSG별로 Flow Log 설정 필요 | VNet 단위로 간편 설정 | | 저장 옵션 | Storage, Log Analytics | Storage, Log Analytics, Event Hub 모두 지원 | | 포맷 | JSON (v1), CSV (v2) | JSON 기반 | | 추가 정보 | NSG 규칙만 기록 | VNet Manager 정책, 암호화 상태 포함 | | 향후 계획 | 2025년 6월 이후 신규 생성 중단 예정 | 신규 대체 기술로 권장됨 |

    📚 공식 문서: VNet Flow Logs 소개

    ⚙️ 구성 방식

    • VNet 단위로 설정하며, 모든 서브넷/NSG의 흐름을 포괄
    • 설정 시 목적지: Storage Account / Log Analytics / Event Hub 지정 가능
    • NSG Flow Log는 리소스 개별 설정이 필요했던 것과 달리 전체 가시성 확보 용이
    az network watcher flow-log configure   --location eastus   --resource-group <rg>   --nsg <vnet-name>   --enabled true   --log-version 2   --storage-account <sa-name>

    ✅ 장점

    • VNet 전체 흐름 기반 분석 가능 → 보안 탐지 및 최적화에 유리
    • 암호화 상태, 정책 기반 흐름까지 기록 가능
    • 구성 간소화 → 설정 누락 및 누수 방지

    ❌ 단점

    • 상대적으로 비용 증가 가능성
    • 많은 양의 데이터 → Log Analytics 요금 폭증 주의
    • Preview 단계에서 일부 기능 제약 존재 가능

    🔐 보안 및 운영 활용

    • Zero Trust 기반 트래픽 분석 가능 (내부 흐름 확인)
    • 이상 징후 탐지, 접근 탐색 분석
    • NVA, 방화벽 등과 연계한 흐름 분석 가능

    ✏️ 시험 포인트 요약 (초압축)

    • 기존 NSG Flow Logs는 2025년 이후 신규 생성 불가 예정
    • 신규 VNet Flow Logs는 전체 흐름 기록, 암호화/정책 포함, EventHub 연동 가능
    • 보안 가시성 및 트래픽 최적화에 유리하나 비용 고려 필요

23. AKS with GPU 구성시, 자원을 효율적으로 사용하기 위한 방안을 서술하세요 문제

  • 2문제 나올것같음

    1. 컨테이너상에서 request/limit 등 qos관련된 내용 알아야하는 문제 burst, best effort, guarantee 3가지가 있다.

    2. gpu는 shared가 불가한거 가능한거 아키텍쳐가 다름(예전 t100 등)

    • 컨테이너가 아무리 나눠서 요청해도 같이 쓸수가 없다.
    • 최신 아키텍쳐는 된다는것 같음.

  • AKS에서 GPU 리소스를 사용하면 고성능 연산(예: AI, 딥러닝)을 효율적으로 수행할 수 있습니다. GPU 노드는 일반 CPU 노드와 별도로 관리되며, 스케줄링 시 GPU를 사용하는 Pod에만 할당됩니다.

    🧠 QoS (Quality of Service) 관련 개념

    🔹 Kubernetes QoS 클래스

    컨테이너의 리소스 request 및 limit 설정에 따라 다음 세 가지 QoS 클래스로 분류됩니다:

    | | | | | ---------- | ----------------------------------- | ----------------------------------------------------------- | | Guaranteed | 가장 높은 안정성 보장 | 모든 컨테이너에 request == limit 이 모두 정의되어 있어야 함 | | Burstable | 일정 수준의 자원 보장 + 버스트 허용 | 일부 request/limit 정의 | | BestEffort | 최소 보장, 버스트 가능 | request/limit 모두 없음 |

    GPU 자원은 반드시 nvidia.com/gpu label로 request 및 limit를 명시해야 사용 가능 Guaranteed 클래스로 분류되어야 안정적인 GPU 스케줄링 가능

    ⚙️ 구성 방식

    • GPU 노드는 전용 Node Pool로 분리하여 운영 (예: NCasT4_v3, ND_v2)
    • taints + tolerations 로 GPU 전용 Pod만 해당 노드에 스케줄링
    • Pod spec 예시:
    resources:  limits:    nvidia.com/gpu: 1  requests:    nvidia.com/gpu: 1

    ⚠️ 제약조건 및 아키텍처 이슈

    🔸 GPU 공유 불가 아키텍처

    • 예전 아키텍처 (예: T100, K80 등)는 GPU Sharing 불가
    • 1 Pod에 1 GPU 단위로 할당 → 낭비 발생 가능

    🔸 최신 GPU (Multi-instance GPU: MIG)

    • A100, H100 등의 최신 NVIDIA GPU는 MIG를 통해 공유 가능
    • Kubernetes와 연동하려면 GPU Operator 및 MIG 설정 필요

    시험 포인트: GPU는 기본적으로 공유 불가능, 최신 GPU 아키텍처에서만 MIG 기능으로 공유 가능

    ✅ 장단점

    | | | | --------------------------- | ---------------------------- | | 고성능 병렬 처리 | 비용 높음 | | 전용 노드 운영으로 안정성 ↑ | GPU 미사용 시 자원 낭비 우려 | | 최신 GPU는 MIG로 효율 ↑ | 복잡한 설정 필요 (MIG) |

    🔐 보안/운영 고려사항

    • GPU 드라이버 및 CUDA 환경 자동 구성은 NVIDIA Device Plugin 필요
    • GPU 노드 오토스케일링은 taint 전략과 cluster autoscaler 조건을 고려해야 함

    📚 공식 문서

    • AKS GPU 노드 구성
    • NVIDIA Kubernetes Device Plugin
    • MIG & Multi-Instance GPU

    ✏️ 시험 포인트 요약 (초압축)

    • request/limit 설정 → QoS에 영향, GPU는 Guaranteed 클래스 권장
    • GPU는 기본적으로 공유 불가 → MIG 있는 GPU만 가능
    • GPU 노드는 taint/toleration 사용, autoscaler 연동 필요
